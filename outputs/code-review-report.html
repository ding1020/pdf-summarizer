⚠️ 发现的问题

🔴

无邮箱验证流程

高危

任意邮箱可完成注册，无需确认。攻击者可用他人邮箱注册，造成账号抢占。

文件：app/api/auth/sign-up/route.ts

建议：注册后发送验证邮件，未验证账号限制登录；或至少发送欢迎邮件确认所有权。

🔴

密码重置 Token 未哈希存储

高危

若重置 token 以明文存入数据库，一旦 DB 泄露，攻击者可直接用 token 重置所有用户密码。

文件：app/api/auth/forgot-password/route.ts（需确认实现）

建议：存储 SHA-256(token)，原始 token 只发给用户邮箱，不落库。

🟠

shareId 缺少 @unique 约束

中危

Document.shareId 仅有 @@index([shareId])，非 @unique。极低概率下两文档拥有同一 shareId，导致共享链接对应错误文档。

文件：prisma/schema.prisma → Document 模型

建议：改为 shareId String? @unique，并在生成时增加碰撞检测。

🟠

webhook 路由 as any 类型绕过

中危

updateUserPro 和 updateUserFree 使用 as any 绕过 Prisma 类型检查，若数据库 schema 变更，编译器无法发现错误。

文件：app/api/webhooks/creem/route.ts，lib/db.ts

建议：明确定义 Prisma update 的类型，移除 as any。

🟠

vercel.json 生产用 db push 而非 migrate deploy

中危

npx prisma db push --skip-generate 在生产中不安全，会直接对 schema 做破坏性变更且无迁移历史，团队协作时有数据丢失风险。

文件：vercel.json → buildCommand

建议：改为 prisma migrate deploy && prisma generate && next build，并维护 migrations/ 目录。

🟠

上传 API 返回完整 content 给前端

中危

/api/upload 将完整 PDF 文本内容（最多 100,000 字符，约 100KB+）返回给前端，再由前端发到 /api/summarize。这导致：1）大文件时网络双倍传输；2）内容在浏览器内存暴露。

文件：app/api/upload/route.ts，components/FileUpload.tsx

建议：上传后返回 documentId，摘要接口直接从 DB 读取 content，避免内容经过客户端。

🟡

CSP 包含 unsafe-inline

低危

Content-Security-Policy 中包含 'unsafe-inline'，降低 XSS 防护级别。

文件：next.config.mjs

建议：迁移到 nonce-based CSP，配合 Next.js 15 的 nonce 支持。

🟡

clerkId 字段语义混乱

低危

自研注册时设置 clerkId: "native_xxxx_xxxx"，此字段原为 Clerk 外部认证设计，现混用为内部 ID，维护时易误解。

文件：app/api/auth/sign-up/route.ts

建议：将字段重命名为 externalId 或 authProvider，并在 schema 中加注释说明用途。

🟡

遗留 Paddle 字段未清理

低危

schema 中含 paddleCustomerId、paddleSubscriptionId、paddlePlanId（标注 deprecated），浪费存储且增加认知负担。

文件：prisma/schema.prisma

建议：用迁移文件删除这些字段，保持 schema 干净。

🔵

完全没有测试

建议

无任何测试文件（单元/集成/e2e），支付逻辑、限流、认证等核心流程没有回归保护，重构风险极高。

建议：优先为 lib/auth-token.ts、lib/rate-limit.ts、app/api/webhooks/creem 编写单元测试（Vitest + supertest）。

🔵

Dashboard Guest Banner 始终显示

UX 建议

dashboard/page.tsx 中 GuestBanner 不区分登录状态，已登录用户也会看到"注册获取更多"的提示，影响用户体验。

建议：将 isSignedIn 状态传入 GuestBanner 组件，已登录时隐藏或替换为 Pro 升级提示。

🔵

重复上传路由

架构建议

同时存在 /api/upload/route.ts 和 /api/documents/upload/route.ts，职责重叠，维护时容易出现不一致。

建议：合并为单一上传路由，或明确区分各自职责并加注释说明。
